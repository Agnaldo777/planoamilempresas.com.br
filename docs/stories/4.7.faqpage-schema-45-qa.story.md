## Story

**As a** decisor PJ buscando respostas operacionais sobre Amil empresarial e **@dev (Dex)**,
**I want** componente `<FAQAccordion />` reusável que renderiza Q&A com schema `FAQPage` JSON-LD válido em todas pillar/cornerstone/programmatic pages, com mínimo 45+ Q&A canônicos,
**so that** o satélite capture rich results SERP (drop-down "People Also Ask"), respeite o anti-erro identificado em `planodesaudeamil.com.br` (45+ Q&A em texto puro sem schema = desperdício de oportunidade) e cumpra FR36 do PRD v1.3.1.

---

## Strategic Context / Why

`planodesaudeamil.com.br` (research `08-acertos-erros-detalhado.md`) tem **45+ Q&A em texto puro** mas zero schema FAQPage — desperdiça rich results em SERP. `amilsaudebr.com.br` (research `09-amilsaudebr-kitcorretor-deep-scrape.md`) tem apenas 4 Q&A com schema — gap explorável. Capturar **45+ Q&A com schema válido** é low-hanging fruit técnico que combina três wins simultâneos: (a) People Also Ask + drop-down em SERP (CTR uplift potencial); (b) cobertura de keywords long-tail conversational; (c) compliance NFR21 (Schema Coverage Mínimo).

Sem componente reusável, cada cornerstone reimplementa schema → drift garantido. Com `<FAQAccordion />`, há single source of truth + testes + validação CI via `schema-validation.yml` (Story 1.3 AC4).

## Acceptance Criteria

1. Componente React Server Component `<FAQAccordion />` em `src/components/seo/FAQAccordion.tsx`:
   - Props: `items: { question: string; answer: string }[]`, `headingLevel?: 'h2' | 'h3'` (default h2), `id?: string`
   - Renderiza HTML semântico: `<dl>` ou `<details>`/`<summary>` (decisão UX em sub-task) com aria attributes corretos
   - Gera JSON-LD `FAQPage` inline via `<script type="application/ld+json">` no mesmo componente — **não duplica em `<Head>`** (RSC compatível)
   - Schema: `{"@context":"https://schema.org","@type":"FAQPage","mainEntity":[{"@type":"Question","name":"...","acceptedAnswer":{"@type":"Answer","text":"..."}}]}`

2. **Catálogo de 45+ Q&A canônicos** em `src/content/faq/faq-amil-empresarial.ts` (TypeScript const, não Sanity inicialmente para manter velocity), distribuídos em 6 categorias:
   - **Carências e CPT** (8 Q&A — ex: "Qual a carência para parto?", "CPT é cobertura?")
   - **Coparticipação e reajuste** (8 Q&A — ex: "Como funciona 30% vs 40%?")
   - **Rede credenciada e abrangência** (8 Q&A — ex: "Posso usar fora do estado?", "Hospitais cobertos?")
   - **Adesão, MEI, sócio único** (7 Q&A — ex: "MEI pode contratar?", "Sócio único cobre como PME?")
   - **Produtos Amil** (8 Q&A — ex: "Diferença entre S750 e S380?", "Black tem cobertura internacional?")
   - **Documentação, contrato, ANS** (6+ Q&A — ex: "Quais documentos para PME?")

3. **Cada answer ≥80 palavras** (não one-liner) — protege contra "thin schema" Google rejection.

4. **Cada answer cita fonte regulatória ou link interno** quando aplicável: ANS RN 195/2009, RN 593/2024, link a cornerstone do tópico, link a glossário (se Story 3.20 done).

5. **Validação Schema.org via CI:**
   - `scripts/validate-schema.mjs` (já existente Story 1.3 AC4 — extender) inclui validação FAQPage especificamente
   - Cada página com `<FAQAccordion />` é renderizada em build e schema extraído + validado contra schema.org JSON Schema
   - Fail build em schema malformado

6. **Validação Google Rich Results Test (manual):**
   - Story DoD inclui screenshot do Rich Results Test mostrando "FAQPage detected" em pelo menos 3 URLs:
     - 1 cornerstone (ex: `/carencia-amil-empresarial/`)
     - 1 pillar (`/`)
     - 1 programmatic landing (sample CNAE×cidade da Wave 1)

7. **Variação contextual em programmatic landings (NFR25 alinhamento):**
   - `<FAQAccordion items={...}>` em programmatic landing recebe **5 Q&A específicas do CNAE/cidade/porte**, não os 45 globais
   - Q&A geradas via template helper `getProgrammaticFAQs(cnae, cidade, porte)` que interpola dados reais (rede regional, faixa etária comum no segmento)

8. **Acessibilidade WCAG 2.1 AA:**
   - Navegação por teclado (Enter/Space toggle accordion)
   - ARIA `aria-expanded`, `aria-controls`, `role="region"` quando expandido
   - Foco visível
   - Screen reader test (NVDA/VoiceOver) — opcional mas documentado em `docs/qa/accessibility-checklist.md`

9. **Tests unitários** `src/components/seo/__tests__/FAQAccordion.test.tsx` (Vitest + Testing Library):
   - Render com 5 itens → schema JSON-LD válido extraído
   - Toggle accordion mantém schema (não re-renderiza só client-side)
   - 0 itens → não renderiza schema (early return)
   - Item com aspas duplas no answer → escape correto no JSON-LD

10. **Lighthouse SEO ≥95** em página com `<FAQAccordion />` (target já é o do projeto; story precisa não regredir).

11. **Imports absolutos** `@/components/seo/FAQAccordion` (NFR14, code-standards.md).

12. **Footprint client JS:** componente RSC = 0KB client JS; accordion interativo via `<details>` nativo (zero JS) OU client island com <2KB.
