## Story

**As a** redator de conteúdo médico, **corretor (revisor técnico)**, **advogado (revisor jurídico)** e **@po**,
**I want** pipeline editorial documentado + Sanity workflow status field + CI gate que bloqueia merge de qualquer conteúdo regulatório/médico sem trilha completa de revisão humana arquivada,
**so that** o satélite cumpra NFR23, evite riscos regulatórios ANS e construa diferencial E-E-A-T verificável vs concorrentes que publicam conteúdo IA-only sem revisão.

---

## Strategic Context / Why

NFR23 proíbe IA-only em conteúdo médico/regulatório (carência, CPT, ANS, reajuste, coberturas, doenças). Atualmente o pipeline não tem **enforcement técnico** — risco de redator publicar via Sanity sem passar por advogado, especialmente sob pressão de cadência editorial. A story converte política em **gate técnico**:

1. Sanity workflow obrigatório (não publish até `reviewedByCorretor && reviewedByAdvogado`)
2. CI valida frontmatter `reviewedBy` em MDX/JSON antes de merge
3. Audit log mensal arquivado documentando quem revisou o quê e quando

Sem este enforcement, NFR23 é aspiracional; com ele é rastreável e auditável (importante para compliance ANS pós-fato e defesa jurídica em caso de disputa). Concorrentes (todos os 3 diretos) não têm processo análogo — diferencial E-E-A-T legítimo.

## Acceptance Criteria

1. **Sanity v3 schema field** em `sanity/schemas/cornerstone.ts` (e `blog.ts`, `programmatic.ts` quando aplicável):
   - Field `reviewStatus` (string options: `draft`, `redator-done`, `corretor-reviewed`, `advogado-reviewed`, `published`)
   - Field `reviewedByCorretor: { reviewer: reference Author, date: datetime, notes: text, approved: boolean }`
   - Field `reviewedByAdvogado: { reviewer: reference Author, date: datetime, notes: text, approved: boolean }` — obrigatório se `category in [carencia, cpt, ans, reajuste, coberturas, doencas]`
   - Validation rule custom: bloqueia transição `published` se algum review faltante para categoria YMYL

2. **Custom Sanity Workflow plugin** (`sanity/plugins/medical-content-workflow.ts`):
   - UI mostra status atual + próximos passos
   - Botão "Solicitar revisão corretor" envia notificação (email ou Slack via webhook)
   - Botão "Solicitar revisão advogado" idem
   - Histórico de transições visível na sidebar

3. **MDX frontmatter equivalent** (para conteúdo em `src/content/` git-based, fallback Sanity):
   ```yaml
   reviewedBy:
     corretor: { id: "agnaldo-silva", date: "2026-04-28", approved: true }
     advogado: { id: "advogado-x", date: "2026-04-29", approved: true, notes: "..." }
   category: carencia # YMYL
   ```

4. **CI gate** `scripts/validate-content-reviews.mjs`:
   - Parsa todos `.mdx`/`.md` em `src/content/` + JSON Sanity exports
   - Para conteúdo categorizado YMYL (carencia/cpt/ans/reajuste/coberturas/doencas): falha build se `reviewedBy.corretor.approved !== true` OU `reviewedBy.advogado.approved !== true`
   - GitHub Action `.github/workflows/validate-content-reviews.yml` em todo PR
   - Bloqueia merge

5. **Audit log mensal** `scripts/generate-review-audit.mjs`:
   - Roda dia 1 de cada mês
   - Lista todos conteúdos publicados no mês + reviewers + datas + categoria
   - Output `docs/audit-content-reviews-2026-MM.md` arquivado em git
   - Cron via GitHub Actions (paralelo ao Story 7.10 e Story 5.7)

6. **Dashboard interno** (Sanity Studio custom view):
   - Filtro: "Pendentes de revisão corretor" / "Pendentes advogado" / "Atrasados >7 dias"
   - Notificação visual quando review SLA estoura (Story 1.0 AC6: 48h advogado conteúdo, 24h emergência)

7. **Doc canônico** `docs/editorial/cms-guide.md` (Story 3.1 deliverable, esta story APPENDS):
   - Seção "Pipeline de Revisão YMYL" com fluxograma:
     ```
     Redator escreve → status: redator-done
       ↓
     Corretor revisa (técnico) → reviewedByCorretor.approved = true
       ↓
     Advogado revisa (jurídico) → reviewedByAdvogado.approved = true
       ↓
     @po publica → status: published
     ```
   - SLA documentado (alinhado Story 1.0 AC6: 48h conteúdo, 24h emergência)
   - Lista de categorias YMYL que **exigem** advogado vs categorias gerais que precisam só corretor

8. **Excepcionalidades documentadas:**
   - Conteúdo "operacional" (changelog, atualizações de tabela de preço sem mudança regulatória) NÃO precisa advogado, apenas corretor
   - Conteúdo de "carência zero" / promessa de cobertura → revisão tripla (corretor + advogado + DPO)
   - Documentar em `cms-guide.md` quem decide categoria

9. **Tests unitários** `scripts/__tests__/validate-content-reviews.test.ts`:
   - 5 fixtures (YMYL com revisão completa = ok; YMYL sem advogado = fail; non-YMYL sem advogado = ok; YMYL com `approved: false` = fail; corrupted frontmatter = fail)

10. **Backwards compat:** conteúdo legado (Stories 5.0 reaproveitamento Bradesco→Amil) recebe categoria + reviewedBy retroativamente em uma migração one-shot — script `scripts/backfill-reviews-legacy.mjs` (manual, com confirmação @po).

11. **Compliance docs cross-link:** `docs/legal/compliance-checklist.md` (Story 1.0 AC6) referencia esta story como mecanismo de enforcement de revisão regulatória ANS.

12. **Notification channel** (Slack ou WhatsApp Business via webhook) configurado para corretor + advogado receberem ping automático em transição "redator-done" → assigned reviewer.

- Aprovação multi-advogado (escopo: 1 advogado oficial)
- Translation review (escopo Brasil-only NFR17)
