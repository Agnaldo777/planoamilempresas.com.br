## Story

**As a** @devops,
**I want** migrar o Sanity Studio do route Next.js `app/(studio)/studio/[[...tool]]/page.tsx` para hosting gratuito Sanity (`https://planoamilempresas.sanity.studio` via `npx sanity deploy`),
**so that** o main site rode 100% em Cloudflare Workers (ADR-011) sem incompatibilidades de runtime — Sanity Studio é incompatível com Workers (depende de APIs Node.js + bundle pesado), e o stakeholder escolheu **Opção A (Sanity hosted gratuito)** em decisão D de 2026-04-29.

---

## Strategic Context / Why

A migração para Cloudflare Workers (ADR-011 Accepted, 2026-04-29) revelou que o **Sanity Studio embedado** em `app/(studio)/studio/[[...tool]]/page.tsx` é incompatível com Workers runtime:

- Studio depende de bundle Node.js completo (autenticação OAuth, presence/realtime, image pipeline)
- Workers tem limites de memória/CPU/bundle size que Studio excede
- OpenNext não consegue adaptar todas as APIs internas do Studio

Stakeholder analisou três opções:
- **Opção A:** Sanity hosted gratuito (`*.sanity.studio`) — escolhida ✅
- Opção B: Subdomínio próprio com Sanity hosted (`studio.planoamilempresas.com.br` via CNAME) — pós-MVP
- Opção C: Vercel-only para `/studio` (custo + complexidade duplo-deploy) — rejeitada

**Opção A** é gratuita, oficial, sem manutenção de infra, e separa concern do main site. URL `https://planoamilempresas.sanity.studio` é gerenciada pela Sanity Inc.

### Key Technical Context

**Por que hosted Sanity (Opção A) e não Vercel-only para /studio:**
- Mantém main site 100% em Cloudflare Workers (objetivo ADR-011)
- Zero custo (Sanity Free tier inclui hosting do Studio)
- Zero manutenção de infra (Sanity gerencia uptime, CDN, certificados)
- Separação de concern (Studio é tool interno; main site é público)

**Sobre `npx sanity deploy`:**
- Builda o Studio com schemas de `sanity.config.ts` + `sanity.cli.ts`
- Sobe para infra Sanity (CDN global)
- Gera URL `https://<hostname>.sanity.studio`
- Cada `npx sanity deploy` subsequente atualiza schema (sem downtime perceptível)
- Login do usuário é Sanity OAuth (Google/GitHub/email)

**CORS é crítico:** sem `Allow credentials: true` nas origins do main site, queries autenticadas (preview mode, draft viewing) falham.
