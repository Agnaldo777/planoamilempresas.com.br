## Story

**As a** `@devops (Gage)` mantendo a infraestrutura editorial e **@pm (Morgan)** garantindo cadência consistente,
**I want** pipeline CI/CD editorial completo: webhook Sanity → Vercel revalidate, GitHub Action de cadência (alerta sem post novo em 14 dias), validação de schema BlogPosting + Article no CI, audit Author YMYL (NFR22) e sitemap auto-update via webhook,
**so that** o blog opere como produto editorial: novos posts publicados aparecem em produção em <1min sem deploy manual, gaps de cadência são detectados automaticamente, e qualquer post YMYL sem author com credenciais é rejeitado em CI antes de ir ao ar.

---

## Strategic Context / Why

Sem auto-publish + revalidate, a equipe editorial depende de deploy manual a cada post — gargalo operacional. Sem alerta de cadência, o blog pode silenciosamente ficar parado (sintoma do anti-padrão `amilplanos.com.br`). NFR22 (Author YMYL) precisa enforce automatizado em CI — não bastam revisão humana, precisa gate técnico que falhe se post de saúde for publicado sem author com credenciais. Esta story fecha o ciclo editorial transformando-o em sistema observável e self-healing.

Pode rodar em paralelo a 6.11.b/c/d porque sua entrega é infraestrutural — não depende do conteúdo das Waves, só do schema/template (6.11.a).

## Acceptance Criteria

1. **Sanity webhook → Vercel revalidate**:
   - Sanity Studio configurado com webhook em mudança de documento `blogPost` (publish, unpublish, update)
   - Endpoint `app/api/revalidate/route.ts` (Next 16) recebe POST do webhook:
     - Valida signature/secret (env `SANITY_WEBHOOK_SECRET`)
     - Faz `revalidatePath('/blog')`, `revalidatePath('/blog/[slug]', 'page')` para slug afetado, `revalidatePath('/blog/categoria/[categoria]', 'page')` para categoria afetada
     - Retorna 200 com revalidated paths em JSON
     - Retorna 401 se signature inválida
   - Latência target: <60s entre publish no Sanity e visibilidade em produção

2. **GitHub Action `editorial-cadence.yml`**:
   - Schedule: cron diário (ex. `0 9 * * *` — 09:00 UTC)
   - Query Sanity (via API token read-only) por último post publicado (`orderBy publishedAt desc, limit 1`)
   - Calcula `daysSinceLastPost = (now - lastPost.publishedAt) / day`
   - Comportamento:
     - `daysSinceLastPost < 14` → success, log "cadence ok"
     - `14 ≤ daysSinceLastPost < 30` → success com warning, abre/atualiza GitHub issue label `editorial-cadence-warning`
     - `daysSinceLastPost ≥ 30` → fail, abre/escala GitHub issue label `editorial-cadence-critical` com @mention `@pm`
   - Issue body inclui: `lastPost.title`, `lastPost.publishedAt`, `daysSinceLastPost`, link para roadmap

3. **Schema validation CI** (extensão do existente `schema-validation.yml`):
   - Adicionar validação de `BlogPosting` em build (rota sample `/blog/[mock-slug]`)
   - Adicionar validação de `Article` (alias) onde aplicável
   - Fail CI se faltar campos obrigatórios: `headline`, `author`, `datePublished`, `image`, `publisher`
   - Validar que `publisher` é Organization BeneficioRH (FR54 — `parentOrganization`)

4. **Author YMYL audit (NFR22)** — gate CI dedicado:
   - Script `scripts/audit-author-ymyl.mjs`:
     - Query todos posts publicados do Sanity
     - Para cada post, verifica:
       - `author` reference existe e resolve
       - `author.name`, `author.credentials` (ex. CRM, OAB, ou título profissional) preenchidos
       - Posts categoria saúde (`Carências`, `Cobertura`, `Coparticipação`, `Reembolso`, `ANS`, `Cancelamento`) → `author.credentials` deve incluir credencial médica/jurídica/técnica reconhecível (regex configurável)
     - Fail CI se algum post YMYL sem author qualificado
   - Roda em PR + main + workflow agendado semanal

5. **Sitemap auto-update via webhook**:
   - Sanity webhook (mesmo de AC1) também triggers regeneração de `app/sitemap.ts` (ou `sitemap.xml/route.ts`)
   - Sitemap inclui todos posts publicados com `lastModified = updatedAt`, prioridade 0.7 para posts, 0.6 para listing/categoria
   - Validação: `scripts/validate-sitemap.mjs` valida XML schema + URLs resolvem 200 (smoke test 3 URLs aleatórias)

6. **Documentação operacional** em `docs/devops/editorial-pipeline.md`:
   - Diagrama (mermaid ou ascii) do fluxo: Sanity → webhook → Vercel revalidate + sitemap
   - Como rotacionar `SANITY_WEBHOOK_SECRET`
   - Como debugar webhook que não disparou (logs Sanity + Vercel)
   - Como ajustar thresholds da GitHub Action de cadência

7. **Observability**:
   - Endpoint revalidate loga estruturadamente (path revalidated, slug, timestamp, source)
   - Métricas básicas: contador de revalidates por dia (via Vercel Analytics ou log query)

8. **Test coverage**:
   - Endpoint revalidate: unit test (signature válida → 200, inválida → 401)
   - Audit YMYL: unit test com fixtures (post válido, post sem author, post YMYL sem credencial → fail esperado)
   - GitHub Action: dry-run em PR validando logic threshold

### Key Technical Context

- **Next 16 revalidate**: `revalidatePath` + `revalidateTag` em route handler. Considerar tag-based para granularidade (`tag('blog-post:slug')`)
- **Sanity webhook signature**: HMAC-SHA256 com `SANITY_WEBHOOK_SECRET`, header `sanity-webhook-signature` (formato `t=...,v1=...`) — usar lib `@sanity/webhook` se disponível
- **GROQ query cadência**: `*[_type=="blogPost" && defined(publishedAt)] | order(publishedAt desc) [0]{title, publishedAt, slug}`
- **Audit YMYL regex credentials**: `/(CRM[\s-]?\d+|OAB[\s-]?\w+|MD|MBA|advogado|médic[oa]|especialista)/i` (configurável em script)
- **GitHub Issues API**: usar `actions/github-script` ou `gh` CLI dentro do workflow para criar/atualizar issue
- **NFR23 enforcement**: este pipeline é o gate técnico — sem ele, revisão humana é o único gate (frágil)
