## Story

## Strategic Context / Why

O dataset 2026-04-26 tem header `redes[]` com **49 strings** mas apenas **11 redes** aparecem populadas em `prestadores[].redes`. Memória `project_amil_rede_credenciada_powerbi.md` registra que o dropdown "Rede" do Power BI tem 176+ produtos — então 49 capturadas é parcial (ou metadado-only). Sem resolver este gap, Story 7.7 (Cluster E SEO — high-conversion) fica com risk de canibalização interna quando as 38 restantes entrarem (cada nova rede pode rebatear URLs já indexadas). Pre-flight isolado preserva o moat. Ver SCP v1.2.3 §3.3 e Pax MAJOR-2-bis.

## Acceptance Criteria

1. Script auxiliar (Node) enumera set-difference entre `dataset.prestadores[].redes` (todas as ocorrências distintas) e `dataset.redes[]` (header). Output: lista de redes em `redes[]` mas zero prestadores.
2. Cross-check com `scrape_powerbi_amil.js` em modo verbose — se necessário, rerun com flag `--all-redes` para forçar enumeração completa.
3. Output documentado em `docs/decisions/network-coverage-2026-04.md` com:
   - 49 redes do header listadas
   - 11 redes ativas (com prestadores) destacadas
   - 38 redes "fantasma" categorizadas: (a) metadados de dropdown sem prestadores, (b) prestadores não capturados pelo scraper
4. **Decisão output:**
   - Output 1 (38 são metadados): GO Story 7.7 com 11 URLs Cluster E (1 por rede ativa)
   - Output 2 (38 têm prestadores): HOLD Story 7.7 + criar Story 7.0a-bis para scraper rerun completo
5. Cross-reference das **11 redes ativas** com **6 segmentações Bronze→Platinum Mais** da `data/tabelas-amil.ts`:
   - Mapeamento explícito (ex: "AMIL S750 QP" → "Ouro" / "AMIL S380 QP" → "Bronze Mais"?)
   - Flag faltante onde não há mapping óbvio
   - Stakeholder Agnaldo confirma mapping em handoff WhatsApp/email
6. Documento publicado com Conventional Commit `docs(decisions): document network coverage analysis [Story 7.0a]`
