## Story

**As a** SEO programmatic strategist,
**I want** 9.325 páginas estáticas `/rede/[uf]/[municipio]/[prestadorSlug]` geradas via SSG com chunking por região (Sudeste em Phase 1; Sul/CO/NE/Norte em Phase 2), schemas markup `MedicalOrganization`/`Hospital`/`MedicalClinic`/`MedicalLaboratory`/`EmergencyService` por tipo inferido e ISR 30d,
**so that** capturamos long-tail prestador-by-name (~9.325 search queries únicas) sem estourar o build budget de 25min do Hobby tier, com noindex anti-thin em cidades < 5 prestadores e schema rich snippet para diferenciação SERP.

---

## Strategic Context / Why

Story 7.4 é o **Ataque 2** do Epic 7 e o de maior **volume bruto** dos 5 (9.325 URLs). Três motivações que ditam o design:

1. **Build budget = constraint dominante.** ADR-005 v2 + SCP §6.1 recalibraram o volume de 5.000-6.500 (plano antigo de 2.071 prestadores) para ~10.500 URLs total. Hobby tier não comporta SSG full sem chunking. A decisão de Phase 1 (Sudeste = RJ+SP+MG+ES = 7.166 prestadores ≈ 77% do volume) prioriza onde a densidade SEO + tráfego histórico Brasil já está, deixando Phase 2 (Sul+CO+NE+Norte ≤2.159) para o segundo deploy.
2. **Anti-thin content é mandatório** (PRD §Story 5.7 audit + Helpful Content Update). Cidade com menos de 5 prestadores → noindex no prestador único + canonical para cidade-pai. Sem esse filtro, ~30% das páginas seriam classificadas thin pelo Google e arrastam autoridade do hub.
3. **Schema markup por subtipo é o moat de diferenciação.** Concorrentes (16 mapeados em `competitor-analysis.md`) usam fallback `LocalBusiness` ou nenhum schema. Aplicar `Hospital`, `MedicalClinic`, `MedicalLaboratory`, `EmergencyService` baseado em `tipoInferido` (do loader 7.1) é o que justifica investimento no `inferTipoAtendimento()` 9-categorias da 7.1.

Esta story é **L (6-8d)** — maior do Epic 7 — porque combina chunking (build engineering), schemas (SEO engineering), templates (UI integration com 7.3) e audit (Story 5.7 amostragem estratificada).

## Acceptance Criteria

1. **Rota dinâmica SSG** `app/(marketing)/rede/[uf]/[municipio]/[prestadorSlug]/page.tsx` com `generateStaticParams()` retornando combinações `{uf, municipio, prestadorSlug}`. Phase 1 retorna apenas Sudeste (UFs RJ+SP+MG+ES); Phase 2 (entrega separada) adiciona demais UFs.
2. **Chunking infraestrutural:**
   - Phase 1: `getAllPrestadores().filter(p => ['RJ','SP','MG','ES'].includes(p.uf))` → 7.166 páginas
   - Phase 2: `.filter(p => !['RJ','SP','MG','ES'].includes(p.uf))` → ≤ 2.159 páginas
   - Toggle via env `PHASE_2_ENABLED=true` em `next.config.mjs` (default `false` no MVP)
3. **Schema markup por tipo inferido** via component `<PrestadorJsonLd />` discriminated union (6 subtipos canônicos Schema.org + fallback, expansão SCP §4.3 com `MedicalImagingService` adicional):
   - `Hospital` (tipo === `'Hospital'`)
   - `MedicalClinic` (tipo === `'Clínica'`)
   - `MedicalLaboratory` (tipo === `'Laboratório'`)
   - `EmergencyService` (tipo === `'Pronto-Socorro'`)
   - **`Hospital` com `medicalSpecialty: { @type: 'MedicalSpecialty', name: 'Obstetrics' }`** (tipo === `'Maternidade'`) — Schema.org não tem `Maternity` como `@type` discreto; padrão canônico é `Hospital` + specialty Obstetrics. Validado em https://schema.org/Hospital
   - **`MedicalImagingService`** (tipo === `'Diagnóstico por Imagem'`) — Schema.org canonical `https://schema.org/MedicalImagingService` (subtype de `MedicalBusiness`). Adicionado via amendment MINOR-04 — sai do fallback `MedicalOrganization` para rich snippet mais específico (Google reconhece radiologia/imagem como categoria distinta).
   - `MedicalOrganization` fallback (tipos `'Outro'`, `'Centro/Instituto'`, `'Odontologia'`)
   - **Type safety:** discriminated union em TypeScript usa `tipoInferido` como discriminant; branch de Maternidade reusa branch `Hospital` com prop `medicalSpecialty` adicional. **6 branches concretos** (Hospital, MedicalClinic, MedicalLaboratory, EmergencyService, MedicalImagingService, MedicalOrganization) + 1 prop opcional `medicalSpecialty` no branch Hospital. Test: parse JSON-LD de fixture maternidade → `@type === 'Hospital'` E `medicalSpecialty.name === 'Obstetrics'`; fixture diagnóstico imagem → `@type === 'MedicalImagingService'`.

3-bis. **`<OrganizationJsonLd />` reuso transversal — ADR-006 mitigação 2 (NOVO via amendment O-2):**
   - Importar `<OrganizationJsonLd />` de `@/components/schema/OrganizationJsonLd` (criada em Story 7.2)
   - Renderizar em **todas as 9.325 páginas-prestador** (Phase 1 + Phase 2)
   - Render usa props default canônicos (`name: 'BeneficioRH (corretor)'`, `sameAs: ['https://www.amil.com.br']`) — **não override** nesta story
   - **Razão:** ADR-006 Context (linhas 22-24) declara que mitigações se aplicam a *"URLs UF/município/bairro (~10.000+ URLs)"* — não só Cluster E. Páginas-prestador renderizam termo "Amil" implicitamente (rede credenciada) e UF/município em URL → mitigação 2 aplicável.
   - **Test:** smoke E2E em 5 URLs estratificadas (AC11) verifica presença de `<script type="application/ld+json">` com `@type: "Organization"` e `sameAs: ["https://www.amil.com.br"]`.
4. **`generateMetadata({params})`** produz title/description/canonical específicos:
   - Title: `{Nome Prestador} em {Município}/{UF} — Rede Credenciada Amil`
   - Description (~155 chars): inclui bairro, tipo inferido, redes que aceita (top-3)
   - `canonical: /rede/[uf]/[municipio]/[prestadorSlug]/`
   - `openGraph` + `twitter` populados
5. **ISR revalidate** = `2592000` (30 dias) + `dynamic = 'force-static'` para Phase 1 + Phase 2.
6. **NoIndex anti-thin:** se `getPrestadoresPorMunicipio(uf, municipio).length < MIN_PRESTADORES_MUNICIPIO_PARA_PRESTADOR_INDIVIDUAL`, página renderiza `<meta name="robots" content="noindex,follow">` e `canonical` aponta para `/rede/[uf]/[municipio]/` (página-cidade da Story 7.5).
   - **Threshold em `src/config/seo.ts`** (centralizado para tuning pós-launch sem code change):
     ```ts
     export const MIN_PRESTADORES_MUNICIPIO_PARA_PRESTADOR_INDIVIDUAL = 5
     ```
   - **Por que 5 (não 3):** município com 1-4 prestadores indica cidade-pequena onde página-prestador individual canibaliza a página-cidade (Story 7.5). Threshold mais alto que `MIN_PRESTADORES_BAIRRO=3` (Story 7.6) e `MIN_PRESTADORES_TIPO=3` (Story 7.8) é proposital — escopo de "prestador individual em cidade pequena" é mais restritivo que escopo de "bairro" ou "tipo agregado".
   - **Test fixture obrigatório:** município com 4 prestadores → noindex + canonical correto; município com 5 prestadores → indexável padrão.
7. **Conteúdo da página** consome templates da Story 7.3 (`<HospitalContent />` / `<LaboratorioContent />` / `<ClinicaContent />` / `<OutroContent />`) para garantir ≥600 palavras únicas + variação contextual ≥8 interpolações nome+bairro+UF (validação Story 5.7 audit).
8. **Sitemap shard** `app/sitemap-prestadores.xml/route.ts` com `<lastmod>` real = `dataset.geradoEm`. Adicionado ao sitemap index principal. Phase 2 expande shard quando ativo.
9. **Internal linking obrigatório:**
   - Breadcrumb: Home › Rede Credenciada › {UF} › {Município} › {Prestador}
   - `<PrestadoresProximos />` (Story 7.3) listando 3 cards do mesmo bairro
   - Link "Ver toda rede em {Município}" → `/rede/[uf]/[municipio]/` (Story 7.5)
   - Link "Outras cidades de {UF}" → drawer com top-10 do estado (mesmo `<UfShortcutChips />` reusado)
10. **Build time targets (NFR2 + budget Vercel):**
    - Phase 1 build ≤ **25 minutos** no Hobby tier (validado via `time pnpm build` em CI matrix)
    - Phase 2 build separado (deploy próprio) ≤ 15 minutos (volume menor)
    - Falha o gate se Phase 1 > 30min
11. **Performance gates (NFR1):** amostra de 10 páginas-prestador aleatórias estratificadas (top-5 cidades + 5 tail) em Lighthouse CI: Perf ≥92, LCP ≤2.0s, INP ≤200ms, CLS ≤0.05.
12. **Disclaimer canônico** importado de `src/content/disclaimers/amil-rede.ts` (criado em 7.2) — DRY enforced.
13. **Mapa aproximado (Leaflet/OSM)** com centroide do bairro/município (sem coordenadas por prestador — gap dataset). Aviso: "localização aproximada — ver app oficial Amil para endereço exato". Lazy-load (não conta no LCP).
14. **CI gate:** PR não mergia sem (a) build verde nos 2 phases, (b) lint:strict, (c) tests verdes, (d) Lighthouse CI nos 10 amostrais, (e) Schema.org validator OK em 5 prestadores estratificados (1 por subtipo principal), (f) `noindex` test passando para um caso fixture com município <5 prestadores.

### Key Technical Context

**Chunking strategy (build budget):**
- Phase 1 (Sudeste): RJ+SP+MG+ES = 7.166 prestadores ≈ 77% volume + ~80% search demand histórica
- Phase 2 (resto): ≤ 2.159 prestadores; ativado via flag separado
- Build time empírico esperado: ~3-4ms por página statics + JSON-LD generation (Vercel Hobby benchmarks). 7.166 × 3,5ms = ~25min — no limite. Mitigação: pré-renderizar JSON-LD em script offline, importar como string.

**Schema discriminated union:**
- Tipo `tipoInferido` é regex-based no loader 7.1 (heurística não 100% precisa)
- Casos ambíguos (ex: "INSTITUTO DE OLHOS") caem em `MedicalOrganization` fallback
- Schema *minimum viable*: `name`, `address.addressLocality`, `address.addressRegion`, `areaServed`, `medicalSpecialty` (apenas se tipo = `Clínica` e dataset trouxer; caso contrário omit)

**ISR 30d rationale:**
- Dataset atualiza mensalmente (Story 7.10)
- 30d cobre ciclo + buffer; menor causaria revalidations sem mudança real
- Force-static garante que Vercel CDN serve sem cold start

**Mapa centroide:**
- Sem coordenadas por prestador → centroide do bairro (geocoder Nominatim build-time) ou centroide do município (fallback)
- Lib: Leaflet + OSM (zero cost). MapLibre como alt se Leaflet bundle estourar.
- Render `next/dynamic` com `ssr: false`

**Anti-thin threshold:**
- 5 prestadores escolhido empiricamente (Helpful Content); pode subir para 8 se audit Story 5.7 sinalizar; flag config em `src/config/seo.ts`
