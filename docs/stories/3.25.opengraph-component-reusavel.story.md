## Story

**As a** decisor PJ compartilhando link da tabela Amil no WhatsApp / LinkedIn / Twitter e **@dev (Dex)**,
**I want** componente `<OpenGraph>` reusável em `src/components/seo/` que recebe props `title`, `description`, `image`, `type` (`website` | `product` | `article`) e gera meta tags `<meta property="og:*">` + `<meta name="twitter:*">` correspondentes com fallbacks automáticos contextuais,
**so that** todas páginas indexáveis tenham OG dinâmico contextual por produto/cidade (vs. OG estático único de `amilsaudebr.com.br`), cumpram FR53 do PRD v1.3.1 e maximizem CTR em compartilhamento social.

---

## Strategic Context / Why

`amilsaudebr.com.br` (research `09-amilsaudebr-kitcorretor-deep-scrape.md`) tem OG mas **estático único** — share de SP, MG, RJ, calculadora ou tabela mostra mesma imagem/título. Decisão UX moderna: cada contexto deve ter OG próprio (cidade no título, produto na imagem, etc.) para CTR otimizado.

Sem componente reusável, cada page.tsx reimplementa meta tags via `metadata.openGraph` Next.js — drift garantido. Com `<OpenGraph>` consumido em layouts + override por page → single source of truth, type-safe, testado.

ADR-008 confirma App Router; Next.js 16 App Router já tem Metadata API nativa (`metadata.openGraph` em `generateMetadata()`). Esta story é wrapping + fallbacks contextuais sobre essa API.

## Acceptance Criteria

1. **Helper `<OpenGraph>` (Server Component / Metadata helper)** em `src/components/seo/OpenGraph.tsx`:
   - Não renderiza JSX — exporta função `buildOpenGraphMetadata(opts: OpenGraphOpts): Metadata['openGraph'] & { twitter: Metadata['twitter'] }`
   - Props/opts:
     ```typescript
     interface OpenGraphOpts {
       title?: string         // fallback: page.title (Story 3.22 buildTitle)
       description?: string   // fallback: page.description (FR32)
       image?: string         // fallback: /og-default.jpg
       imageAlt?: string
       type?: 'website' | 'product' | 'article' | 'profile'
       url?: string           // fallback: full URL
       publishedTime?: string // article only
       modifiedTime?: string  // article only
       author?: string        // article only
     }
     ```
   - Saída ready para spread em `metadata.openGraph` Next.js App Router

2. **Fallbacks automáticos contextuais:**
   - `title` ausente → usa `page.title` (resultado do `buildTitle()` Story 3.22)
   - `description` ausente → usa `page.description` (FR32 dinâmica meta description)
   - `image` ausente → usa `/og-default.jpg` (placeholder com logo BeneficioRH + slogan padrão)
   - `imageAlt` ausente → gerado de `title`
   - `url` ausente → usa `process.env.NEXT_PUBLIC_SITE_URL + pathname`
   - `type` ausente → defaults `website`

3. **Twitter Card automático:**
   - `twitter.card`: 'summary_large_image' default
   - `twitter.title`, `twitter.description`, `twitter.image`: espelham OG
   - `twitter.site`: '@beneficiorh' (handle a confirmar Story 1.0 stakeholder)

4. **Imagens dinâmicas para conteúdo programmatic** (opcional v1, target v2):
   - Endpoint `/api/og` (Next.js Route Handler com Edge Runtime + `@vercel/og`):
     - Aceita query params `title`, `subtitle`, `theme`
     - Gera PNG 1200×630 dinâmico em runtime cached
   - Pages programmatic (Story 5.X) e cidade (Story 7.5) usam `/api/og?title=Plano%20Amil%20S%C3%A3o%20Paulo&subtitle=2026` automaticamente
   - Cornerstones e blog (Story 6.11) podem usar imagem custom em frontmatter

5. **Integração em `app/layout.tsx`** (default global):
   - `metadata.openGraph` populated com defaults BeneficioRH
   - Cada page override via `generateMetadata()` retornando `{ openGraph: buildOpenGraphMetadata({...}) }`

6. **Páginas onde override obrigatório:**
   - Homepage: OG com slogan + CTA
   - Tabela preços: OG com imagem da tabela snapshot ou imagem dinâmica `/api/og?...`
   - Cornerstone: OG com `type: article`, author, datePublished
   - Cidade: OG com imagem dinâmica + nome cidade
   - Produto: OG com `type: product`, imagem custom do produto
   - Blog post: OG com `type: article`, author, datePublished

7. **Tests unitários** `src/components/seo/__tests__/OpenGraph.test.ts`:
   - 8 cenários (todos campos preenchidos, todos defaults, mix)
   - Fallbacks consistentes
   - Truncate longo description (≤200 chars)
   - URL absoluta com siteUrl env

8. **Validação manual em PR DoD:**
   - Screenshot Open Graph Debugger Facebook em 3 URLs (homepage, tabela, cornerstone)
   - Screenshot Twitter Card Validator em 3 URLs
   - Documentar em PR

9. **Lint rule custom** `eslint-plugin-local/require-openGraph`:
   - Warn se `generateMetadata()` retorna sem `openGraph`
   - Não bloqueia (gera default), mas alerta dev

10. **NFR11 alignment:**
    - OG title sincronizado com `<title>` (Story 3.22)
    - OG description sincronizado com meta description (FR32)
    - Single source of truth via helpers

11. **Performance:**
    - Helper é build-time (zero runtime overhead em SSG)
    - `/api/og` endpoint Edge Runtime (low latency)
    - Lighthouse SEO ≥95 mantido

12. **Acessibilidade `imageAlt`:**
    - Required em produção (warn em dev se ausente)
    - Default fallback gerado de title é melhor que vazio

13. **Documentação** `docs/seo/opengraph-strategy.md`:
    - Quando usar custom image vs `/api/og` vs default
    - Truncation rules (title 60 char, description 200)
    - Screenshot exemplos OG bem-feito

14. **NFR54 compliance:**
    - OG `type: product` em produtos NÃO declara `Organization.name: "Amil"` no schema (já coberto Story 3.26)
    - `og:site_name` é "BeneficioRH" (não "Amil")

15. **Backward compat:** páginas existentes (Story 1.4 canary metadata) migram para usar helper sem regressão.
