## Story

**As a** @dev (Dex) e **decisor PJ buscando "preço plano amil [estado]" em SERP**,
**I want** schema `AggregateOffer` declarado **por estado** em cada página estadual `/rede-credenciada/[uf]/` e `/precos/[uf]/`, com `lowPrice`, `highPrice`, `offerCount`, `priceCurrency: BRL`, `availability: InStock` derivados do dataset `tabelas-amil.ts` filtrado por UF,
**so that** o satélite cubra FR51 do PRD v1.3.1, vença em granularidade regional vs `amilsaudebr.com.br` (que declara apenas AggregateOffer nacional R$ 102–3500), capture rich results SERP "preço plano amil [estado]" e construa moat de 14+ páginas estaduais com schema rich.

---

## Strategic Context / Why

`amilsaudebr.com.br` declara AggregateOffer **apenas nacional** (R$ 102–3500) — sinaliza ao Google range único válido para qualquer query. Per-estado é gap explorável: SERP "preço plano amil são paulo" e SERP "preço plano amil minas gerais" devem mostrar **ranges diferentes** correspondentes à realidade tabular Amil PME (que varia por UF devido a coparticipação 30% vs 40% e ajustes regionais).

FR51 captura este gap com **derivação dinâmica via dataset** (`tabelas-amil.ts` Story 6.1) — não hardcoded. Mudança de tabela mensal (Story 6.6) automaticamente recalcula AggregateOffer de cada UF; zero drift manual.

Story complementa Story 7.5 (páginas-cidade rede credenciada) — esta cobre **estaduais agregados** com foco em **preço range**, enquanto 7.5 cobre **municípios** com foco em **prestadores**.

## Acceptance Criteria

1. **Helper `buildAggregateOfferByUF(uf: string)`** em `src/lib/seo/aggregate-offer.ts`:
   - Lê `data/tabelas-amil.ts` filtrado por `uf`
   - Calcula `lowPrice` (min priceSpecification em todas faixas etárias × produtos disponíveis em UF)
   - Calcula `highPrice` (max idem)
   - Calcula `offerCount` (count distinto produtos × faixas × coparticipação variants em UF)
   - Retorna objeto Schema.org válido:
     ```json
     {
       "@type": "AggregateOffer",
       "priceCurrency": "BRL",
       "lowPrice": "212.50",
       "highPrice": "1845.00",
       "offerCount": 60,
       "availability": "https://schema.org/InStock",
       "areaServed": { "@type": "AdministrativeArea", "name": "São Paulo" },
       "validFrom": "2026-01-01",
       "validThrough": "2026-12-31"
     }
     ```
   - Fallback: se UF não tem tabela (ex: 6 dos 20 estados ainda a definir Story 6.1), retorna null + warning

2. **Integração em `/rede-credenciada/[uf]/page.tsx`** (Story 7.5 ou similar):
   - `generateMetadata()` ou JSON-LD inline injeta `AggregateOffer` da UF
   - Contexto: schema raiz da página é `MedicalBusiness` ou `WebPage` com `mainEntity` — `AggregateOffer` aparece em `Product.offers` ou diretamente como entity

3. **Integração em `/precos/[uf]/page.tsx`** (Story 6.X):
   - Schema raiz `Product` (Amil PME UF) com `offers: AggregateOffer`
   - PriceTable (Story 6.1) renderizada inline + AggregateOffer derivado dela

4. **Schema validation CI:**
   - `schema-validation.yml` (Story 1.3 AC4 + Story 4.7 / 6.9 extensions) valida AggregateOffer presente em todas `/precos/[uf]/` e `/rede-credenciada/[uf]/` indexáveis
   - Fail build se UF sem AggregateOffer (exceto gracefully UFs sem dataset)

5. **Validação Schema.org Rich Results Test (manual):**
   - DoD inclui screenshot 3 UFs amostradas (SP, RJ, MG) mostrando "Product / AggregateOffer detected"

6. **Tests unitários** `src/lib/seo/__tests__/aggregate-offer.test.ts`:
   - Cenário SP (10 faixas × 6 produtos × 2 copart = 120 offers; lowPrice e highPrice corretos)
   - Cenário UF sem dataset → null
   - Cenário UF com dataset parcial (só Bronze e Prata) → offerCount correto, range correto
   - Edge: produtos premium (Black, Amil One S6500) com `<PrecoSobConsulta>` (sem price) → exclusos do AggregateOffer (não inflam range com 0 ou NaN)
   - Edge: dataset com priceSpecification corrompido → throw com erro claro

7. **NFR21 cobertura schema mínima:**
   - Cada `/precos/[uf]/` indexável tem AggregateOffer válido (CI gate)
   - Página `/precos/` master (nacional) tem AggregateOffer agregado nacional como fallback

8. **Documentação** `docs/architecture/schema-aggregate-offer-strategy.md`:
   - Por que per-estado vence nacional
   - Como manter dataset compatível
   - Quando AggregateOffer deve ser excluído (UF sem cobertura)

9. **Internal linking sinalizando ranges:**
   - Hub `/precos/` master linka cada UF com texto "Tabela Amil [UF] — R$ X a R$ Y" (interpolando AggregateOffer)
   - Lista cidade do hub `/rede-credenciada/` linka UFs com badge de count prestadores + range preço

10. **Performance:**
    - Cálculo AggregateOffer cached em build-time (zero runtime overhead)
    - SSG: invalidação ISR alinhada com Story 6.6 (pipeline mensal tabela)
    - Lighthouse SEO ≥95 mantido

11. **Compliance ANS / disclaimer:**
    - Schema NÃO substitui disclaimer visual: página continua exibindo "Valores sujeitos a análise da operadora..." (NFR6)
    - `validFrom`/`validThrough` cobrem ano corrente (interage com Story 3.22 currentYear)

12. **NFR17 escopo Brasil:**
    - `priceCurrency: "BRL"` único
    - `areaServed.name` em português + `addressCountry: "BR"` quando aplicável

13. **Compatibilidade Story 6.1 PrecoSobConsulta:**
    - Produtos premium não entram em AggregateOffer (não têm price)
    - `offerCount` reflete só produtos com price público
    - Doc explícita esta exclusão para evitar confusão futura

14. **Tests E2E** Playwright: 3 URLs estaduais com schema válido em network response.

## Out of Scope

- AggregateOffer per-cidade (granularidade extrema) — não necessário, estadual basta para SERP
- AggregateOffer cross-operadora — escopo Amil
- Real-time price update via API — dataset estático mensal (Story 6.6)
- Currency multi-moeda — BRL único
- Promoções temporárias com `validThrough` curto — não MVP
