## Story

**As a** @devops,
**I want** GitHub Actions rodando lint, typecheck, testes unitários, Lighthouse CI, axe-core e schema validation em cada PR, com deploy automatizado via Vercel GitHub Integration,
**so that** qualidade, performance e acessibilidade sejam garantidas antes do merge e deploy seja zero-friction.

---

## Strategic Context / Why

Sem CI automatizado, qualquer regressão (perf <90, A11y violation, build broken) só seria detectada manualmente — inviável com volume de stories de Epic 3-5 (15 cornerstones + 600 programáticas + Wave 2/3). Lighthouse CI threshold ≥90 alinhado com NFR1 (≥92 meta) trava merges que comprometeriam SEO. Axe-core trava A11y violations WCAG AA antes de qualquer story de UI ser merged. Esta story é o "guardião" automatizado do Quality First constitutional principle.

---

## Acceptance Criteria

(Fonte: PRD v1.2.2 Epic 1 Story 1.3, 9 ACs)

1. Workflow `ci.yml` rodando em PR: install deps, lint (`eslint-config-next`), typecheck (`tsc --noEmit`), test unit (`vitest run`), build (`next build`)
2. Workflow `lighthouse.yml` rodando Lighthouse CI em PRs com threshold **Performance ≥90** (alinhado com NFR1 ≥92 meta), rodado contra URL de preview Vercel via `lhci-action`
3. Workflow `accessibility.yml` rodando axe-core em componentes isolados + páginas-chave, falhando em violações WCAG AA
4. Workflow `schema-validation.yml` validando schema.org via Google Rich Results Test (utility para stories de conteúdo Epic 3+)
5. Vercel GitHub Integration ativa: deploy automático preview em cada PR; deploy production em merge para `main`
6. Status checks bloqueando merge: CI, Lighthouse, Accessibility, Vercel Preview Deploy, CodeRabbit (se adotado)
7. Badges de CI no README.md (CI, Lighthouse, Accessibility, License)
8. Cache de node_modules via `actions/setup-node@v4` com `cache: 'npm'` para builds <2min
9. Secret management: GitHub Actions secrets sincronizados com Vercel env vars; script documentado em `docs/devops/sync-secrets.md`

## Out of Scope

- E2E tests Playwright em CI — Story posterior (Epic 4 conversion flows)
- Deployment workflows para staging environment dedicado — projeto MVP usa preview Vercel como staging
- Performance budgets além de Lighthouse Perf threshold (ex: bundle size limits) — pode ser adicionado em iteração posterior
- Workflow `deploy.yml` para deploy manual a partir de tag — Vercel cuida via GitHub Integration
- Notification (Slack/Discord/email) em failures — pós-MVP
- CodeRabbit configuration formal (depende de decisão `core-config.yaml`) — preparado mas opcional

### Key Technical Context

**Threshold rationale:**
- **Performance ≥90** (NFR1 alvo ≥92 — passar com folga)
- **A11y ≥95** (WCAG AA target)
- **SEO ≥95** (cornerstone para projeto SEO)
- **Best Practices ≥90**

**Secrets sync:** Cada env var Vercel deve ter mirror em GitHub Actions secrets. Atualização manual via GitHub UI ou `gh secret set`. **NUNCA hardcode** em workflows YAML.
